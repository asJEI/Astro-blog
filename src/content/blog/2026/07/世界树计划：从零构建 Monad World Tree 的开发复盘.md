# 从零构建一棵链上世界树：Monad World Tree 开发全记录

> 一棵树能长多大，取决于扎根有多深。而一棵链上的树，它的根扎在每一个参与者的信念里。

## 写在前面

做这个项目的初衷其实很简单——Web3 世界里有太多的 DeFi、NFT、GameFi，大家都在聊收益、聊铭文、聊打新，但很少有人去做一件"没什么用"却很浪漫的事：在链上种一棵树。

Monad World Tree 就是这样一棵社区共建的世界树。每一个用户留下的 Leaf（留言）都会化为能量，滋养这棵树不断成长。它不会给你带来收益，不会让你暴富，但它会静静生长——因为有一群人相信，有些东西的价值不能用价格来衡量。

这篇文档不是官方技术文档，更像是一次开发过程的完整复盘。从合约设计到前端交互，从视觉动效到用户体验，我会尽量把踩过的坑、做过的取舍、以及为什么这么做的思考都写下来。如果你也想做一个"没那么实用但有点意思"的链上应用，希望这篇文章能给你一点启发。（项目 PRD 在文末）

---

## 整体架构概览

在动手写代码之前，我习惯先用自己熟悉的 Web2 思维去映射 Web3 的体系。毕竟，换了技术栈，产品的底层逻辑是相通的。

如果用 Web2 的视角来看，Monad World Tree 的架构大概是这样的：

| Web2 概念 | Web3 对应  | 说明                                     |
| --------- | ---------- | ---------------------------------------- |
| 后端服务  | 智能合约   | 所有业务逻辑都在链上，公开透明           |
| 数据库    | 区块链状态 | 数据存在链上，任何人都可以读取           |
| 用户账号  | 钱包地址   | 没有注册流程，有钱包就是用户             |
| 登录认证  | 钱包签名   | 用私钥签名来证明身份                     |
| 前端页面  | DApp 前端  | 和 Web2 前端没什么区别，只是数据来源变了 |

具体到我们的项目，技术栈是这样的：

- **区块链**：Monad Testnet（高性能 EVM 兼容链）
- **智能合约**：Solidity + Foundry
- **前端**：React + TypeScript + Vite
- **样式**：TailwindCSS
- **Web3 交互**：wagmi + viem
- **可视化**：纯 SVG + CSS 动画

整个项目有两个核心合约：

1. **WorldTree** — 世界树的核心逻辑，管理等级、能量、总贡献者数
2. **LeafBoard** — 留言板，用户种植 Leaf 的地方，存储所有留言内容

为什么要拆成两个合约？其实是一个关注点分离的考虑——世界树本身是"生命系统"，而留言板是"交互入口"。未来如果想加新的能量来源（比如 NFT 铸造、质押等），只需要调用 WorldTree 的能量增加接口，不用动 LeafBoard 的逻辑。

## 智能合约设计

### WorldTree 合约：一棵会成长的树

世界树的核心机制其实很简单：**能量积累 → 等级提升 → 形态变化**。

```solidity
contract WorldTree {
    struct TreeInfo {
        uint256 level;           // 当前等级
        uint256 totalEnergy;     // 总能量
        uint256 currentEnergy;   // 当前等级的能量
        uint256 contributors;    // 贡献者数量
    }

    mapping(address => bool) public hasContributed;
    uint256[] public levelThresholds; // 各等级所需能量阈值
}
```

这里有几个设计上的取舍想聊一聊：

**为什么用"等级"而不是连续生长？**

从技术上来说，做一个连续生长的树是可行的——能量多少，树就有多大。但我最终还是选择了离散的等级制，原因有两个：

第一，仪式感。升级是一个明确的节点，是整个社区的共同成就。当世界树从"嫩芽"长成"幼树"的那一刻，每个参与者都会有"我也出了一份力"的感觉。这种里程碑式的体验，是连续生长给不了的。

第二，视觉实现成本。五个阶段各做一套精致的视觉效果，远比做一个无限连续的插值要可控得多。与其做一个处处都差点意思的连续动画，不如把五个阶段打磨到位（虽然贴图还是很抽象和糟糕）。

**等级阈值怎么定？**

我用的是指数增长模型——越到后面，升级越难。这很符合现实中的生长规律：种子发芽只需要一点能量，但要长成参天大树，需要的能量是指数级增长的。

```solidity
levelThresholds = [0, 100, 500, 2000, 8000];
```

五个等级，对应五个阶段：种子 → 嫩芽 → 幼树 → 古树 → 世界树。从 0 到 100 能量就能发芽，但要走到最后一阶，需要 8000 能量——这意味着社区必须真正活跃起来，而不是几个人就能撑起来。

### LeafBoard 合约：每一片叶子都是一个故事

LeafBoard 相对直接，就是一个链上留言板：

```solidity
struct Leaf {
    address author;
    string content;
    uint256 timestamp;
    uint256 likes;
}

Leaf[] public leaves;
mapping(uint256 => mapping(address => bool)) public hasLiked;
```

用户种植一片 Leaf，会消耗一点 gas，同时给世界树贡献 5 点能量。

**为什么留言内容存在链上？**

这其实是一个成本和价值的权衡。把文本存在链上很贵（gas 费高），但对于这个项目来说，"每一片叶子都永久刻在链上"本身就是意义的一部分。

想象一下，五年后、十年后，这棵树还在那里，那些早期的留言也还在那里。也许写留言的人早就不记得了，但区块链替他们记着。这种时间的厚重感，是中心化服务器给不了的。

当然，实际部署的时候，我会对留言长度做限制——毕竟，太长的文字既贵又没必要。真正重要的话，几句话就够了。

## 前端的沉浸式体验

做完合约之后，我开始思考前端应该是什么样子。

一开始我做的是普通的 Dashboard 风格——左边一棵树，右边一堆数据卡片，上面是导航栏。功能都有，但总感觉不对：用户打开网站，像是在看一个数据管理后台，而不是进入一个世界。

于是我推翻重做，把整个首页的设计目标从"展示数据"改成了"创造沉浸感"。

### 核心原则：世界树是主角

整个页面只有一个主角，就是世界树。它应该占据视觉中心，所有其他元素都应该是"辅助"而不是"竞争"。

具体来说，我做了这几件事：

1. **树的尺寸放大到视觉主导地位** — 不再是角落里的小图标，而是占据屏幕中央的主体
2. **背景用深绿色调 + 光晕** — 营造一种在森林深处的氛围
3. **数据面板做得很小很克制** — 放在右上角，需要的时候才去看
4. **底部操作栏极简** — 最后只保留了几个简单的按钮

信息架构上做了大幅减法。以前总觉得"这个数据也要展示，那个功能也要入口"，后来发现，少即是多。用户来这里不是为了看报表，而是为了感受一棵正在生长的树。那些详细数据，放进侧边栏就好。

### 侧边栏：一本可以翻开的书

侧边栏我把它叫做"世界树之书"，其实就只是一个指南而已。当用户打开侧边栏，最想知道的通常是"现在怎么样了"，"未来会长成什么样"以及"这东西是什么"。

侧边栏的切换动画我也纠结了很久。一开始用的是简单的淡入淡出，总觉得差点意思。后来改成了从右侧滑入的效果——就像翻书一样，新的一页从右边滑过来。

### 悬浮的叶子消息

这个交互是我比较满意的一个设计。

普通的留言板就是一条一条列出来，很无聊。但如果把每一条留言变成一片漂浮在世界树周围的叶子呢？叶子微微放大发光伴随者呼吸效果，留言的消息就像是一个个悬浮在树木周围的气泡或者是气泡。

当然在实现过程中也踩了几个坑，值得记一下：

**第一个坑：位置随机，刷新就变**

一开始我用的是 `Math.random()` 来生成叶子位置，结果每次刷新页面，叶子全都换地方了。虽然不是 bug，但体验很奇怪——就像你离开一会儿回来，发现整个森林都搬家了。

解决方案是用**确定性伪随机**。根据叶子的索引和当前等级作为种子，用正弦函数生成伪随机数。这样，同一等级下叶子的位置是固定的，刷新不会变。只有等级提升了，叶子的分布才会变化——就像树长大了，周围的叶子也重新分布了。

```typescript
function seededRandom(seed: number) {
  const x = Math.sin(seed) * 10000;
  return x - Math.floor(x);
}
```

**第二个坑：叶子之间互相重叠**

均匀分布的叶子很容易挤在一起，尤其是数量多的时候。我试过环形分布、网格分布，效果都不太好。最后用的是**菲波那契螺旋（黄金角分布）**——就是向日葵籽那种排列方式，天然均匀不重叠。

```typescript
const goldenAngle = Math.PI * (3 - Math.sqrt(5));

for (let i = 0; i < leafCount; i++) {
  const t = (i + 0.5) / leafCount;
  const angle = i * goldenAngle;
  const radius = maxRadius * Math.sqrt(t);
  const x = centerX + Math.cos(angle) * radius;
  const y = centerY + Math.sin(angle) * radius * 0.9;
}
```

用黄金角分布的叶子，会自然形成一种螺旋图案，既均匀又有种数学的美感，很符合世界树的调性。

**第三个坑：消息详情卡和其他 UI 重叠**

叶子飘在屏幕各处，有些靠近边缘，或者被 HUD 数据面板挡住。

解决方案是双层的：

1. **布局时避开 UI 区域** — 右侧的 HUD 面板、底部的进度条、左侧的侧边栏，这些区域都不生成叶子
2. **详情卡智能定位** — 如果叶子在屏幕下半部分，详情卡就往上弹出；如果在上半部分，就往下弹出。这样永远不会超出可视区域

还有一个细节体验：鼠标从叶子上移开后，详情卡片不会立刻消失，而是等 2 秒再消失。如果在这 2 秒内鼠标移到了详情卡片上，计时器就会重置——这样用户就有时间去点击卡片里的"点赞"和"回复"按钮，不会因为手稍微抖一下卡片就没了。

## 世界树的可视化

这可能是整个项目里改得最多的部分。

### 从静态立绘到动态 SVG

一开始我想的很简单：五个等级，画五张图，等级到了就切换图片。我甚至用 AI 生成了五张很精致的立绘——赛博朋克风格的能量种子、电路板纹路的嫩芽、流光溢彩的世界树……

但是放到页面里一看，太丑了。

不是图画得不好，而是**静态的图片放在一个动态的网页里，就像贴上去的贴纸**，没有生命力。而且图片的缩放效果很差，模糊的时候特别明显。

于是我全部推翻，改成了纯 SVG 实现。

SVG 的好处很明显：

- **矢量无损** — 放大多少都不会糊
- **可交互** — 每个部分都可以单独加动画
- **可插值** — 等级之间可以平滑过渡，而不是突兀地切换
- **体积小** — 几 KB 就搞定，比图片快多了

### 五个阶段的生长逻辑

世界树的五个阶段，每个阶段都有自己的视觉特征：

| 阶段   | 形态                | 核心元素                       |
| ------ | ------------------- | ------------------------------ |
| 种子   | 悬浮的能量核心      | 双层光环 + 环绕粒子 + 呼吸脉动 |
| 嫩芽   | 小树干 + 一小团树冠 | 轻微摇摆 + 几点萤火虫          |
| 幼树   | 分叉树枝 + 两层树冠 | 能量流动纹路 + 飘落叶子        |
| 古树   | 粗壮树干 + 茂盛树冠 | 金色光环 + 更多粒子 + 根系     |
| 世界树 | 三层树冠 + 顶部星芒 | 紫色光环 + 完整根系 + 大量粒子 |

每个阶段之间不是突变的，而是通过进度值进行插值过渡。比如当前等级是 2，进度 50%，那么树的大小就在阶段 2 和阶段 3 的正中间。这样能量增加的时候，树是真的在"慢慢长大"，而不是跳一下就变了。

### 那些让树"活起来"的小细节

光有形状还不够，要让一棵树感觉是活的，需要很多细碎的动画：

**呼吸感** — 整棵树（以及背景光晕）在缓慢地缩放，就像在呼吸。频率很慢，大概 4 秒一次，不仔细注意不到，但没有的话就会觉得树是死的。

**随风摇摆** — 树干和树冠有一个非常轻微的左右摆动，幅度只有 1 度左右，模拟风吹的感觉。同样是很细微的效果，但有了它，树就有了重量感。

**能量流动** — 树干里有几条淡淡的光纹在缓缓向上移动，就像树在吸收能量。等级越高，光纹越亮、越多。

**萤火虫粒子** — 树的周围飘着一些发光的小点点，随机地移动、闪烁。这些小东西特别能营造"魔法森林"的氛围，而且实现成本很低。

**飘落的叶子** — 从等级 2 开始，会有叶子从树冠上缓缓飘落。数量不多，就那么几片，但效果特别好——仿佛能感觉到时间在流逝。

做这些细节的时候，我最大的感受是：**好的动效是你感觉不到它在动，但没有它你就会觉得哪里不对**。它应该是潜意识层面的体验，而不是刻意去展示的炫技。

### 踩过的 NaN 的坑

说一个有点好笑但很真实的坑。

刚做完 SVG 版本的时候，控制台里全是 NaN 错误——`Received NaN for the 'rx' attribute`、`Received NaN for the 'cy' attribute`……一排全是。

排查了半天才发现，是因为初始状态下有些值是 undefined，经过计算就变成了 NaN。比如等级 0 的时候，树干高度是 0，树冠半径是 0，用这些值去做插值运算，结果就炸了。

解决方案很简单但很重要：给所有计算都加一个安全检查函数。

```typescript
function safeNum(val: number | undefined | null, fallback: number): number {
  if (typeof val !== "number") return fallback;
  if (isNaN(val) || !isFinite(val)) return fallback;
  return val;
}
```

每一个用到插值结果的地方，都用 `safeNum` 包一层，给一个合理的默认值。从此世界清净了。

这个事情给我的教训是：**只要是计算出来的值，永远不要假设它一定是数字**。尤其是在前端做可视化的时候，一个 NaN 就可能让整个画面崩掉。

## 一些还没做完的事

写到这里，项目其实还远没有完成。还有很多想法在脑子里，没有来得及实现：

- **成就徽章系统** — 比如"第一个种 Leaf 的人"、"种了 100 片叶子"、"见证世界树升级"……这些链上成就，每一个都是独一无二的纪念
- **贡献者图谱** — 用可视化方式展示所有贡献者和他们的 Leaf，让每个人都能在树上找到自己的位置
- **季节变化** — 世界树会随着时间有四季的变化，春天发芽，秋天落叶，增加时间的维度
- **音乐和音效** — 风吹树叶的声音、能量流动的声音，让沉浸感再上一个台阶

有些功能可能永远也不会做，因为优先级确实不高。但没关系，世界树本来就是一个慢慢生长的项目。就像真正的树不会一夜之间长成参天大树，一个好的产品也需要时间慢慢打磨。

## 从这个项目，我重新理解了 Web3 产品开发

写到这里，我忽然发现，这个项目真正让我成长的地方，并不是学会了多少 Solidity，也不是写出了多少动画——而是让我真正开始思考：

> **什么样的产品，才值得放到区块链上？**

以前我一直觉得，Web3 开发就是学习新的技术栈：学钱包、学 Solidity、学部署合约、学链上交互。后来真正做完一个项目以后，我发现，这些其实都只是工具。真正困难的是：**产品设计。**

### Web3 不只是把数据库换成区块链

这是我最大的认知变化。

最开始设计 Monad World Tree 时，我也想过把所有数据都放到链上——留言、点赞、动画状态，甚至世界树的每一个视觉变化。后来我发现，这其实是一种 Web2 思维。

因为真正的问题不是：

> 数据放在哪里？

而应该是：

> **哪些东西值得让所有人共同相信？**

最后我留下来的链上数据其实很简单：

- 世界树当前等级
- 总能量（Energy）
- 每一片 Leaf
- 每一次贡献记录

而动画、SVG、光效、粒子这些体验，则全部留在前端——不是因为技术做不到，而是因为它们本来就不需要共识。

> 数据库可以保存数据。区块链保存的应该是规则。

这是我做完整个项目以后最大的理解。

---

### 创造体验

开始做之前，我觉得 Web3 产品最重要的是钱包、是交易、是金融。后来发现并不完全是。

用户关心 Gas、RPC、Chain ID——同时也有一部分人关心的是：

> 我刚刚种下了一片 Leaf，然后世界树真的长大了一点。

这种反馈，比任何交易成功提示都更有意义。

好的链上产品，不应该让用户一直感受到「我在用区块链」，而应该让用户觉得：

> 我正在参与一个有意思的世界。

区块链只是这个世界运行的方式，而不是产品本身。

---

### AI 改变了我的开发方式

这个项目几乎贯穿了我整个 AI Coding 的学习过程。

以前开发一个功能，我会先去搜索、看文档、找教程、复制代码、再修改。现在很多时候，我更习惯先描述我要做什么。例如，我不会说「帮我写一个 SVG」，而是告诉 AI：

> 我希望世界树像真正有生命一样，会缓慢呼吸，会随着等级成长，会让用户感觉自己的贡献真的改变了它。

AI 会帮我完成代码，但真正决定产品方向的，一直都是我。

后来我越来越觉得：**Prompt 其实也是一种开发能力**——它更像是在训练自己，如何把一个模糊的想法表达成一个可以被执行的需求。

---

### 技术可以复制，产品思考不能

整个 Monad World Tree 项目，没有特别复杂的算法，也没有炫技式的架构。如果把代码放到 GitHub，也许很多人一天就能复刻出来。

但我觉得真正无法复制的，是最开始那个问题：

> 如果世界树真的存在，它应该如何成长？

后来所有的设计，几乎都是围绕这个问题展开的：

- 为什么叫 Leaf？
- 为什么留言会增加 Energy？
- 为什么成长要分阶段？
- 为什么首页不是 Dashboard？
- 为什么世界树是整个页面的主角？

这些答案，没有写在任何教程里。它们都是不断尝试、不断推翻之后慢慢形成的。

我越来越觉得：一个好的 Web3 项目，最终竞争的不是代码，而是**产品想象力**。

---

## 如果重新开始，我会怎么做？

如果让我重新开发一次 Monad World Tree。我不会第一天就打开 Cursor，也不会马上开始写 Solidity。而是会先画一张产品草图，先回答几个问题：

- 用户为什么愿意来到这里？
- 用户完成一次交互以后，会得到什么反馈？
- 哪些状态必须记录在链上？
- 为什么不用普通数据库？
- 如果没有区块链，这个产品还能成立吗？

只有这些问题都有了答案之后，我才会开始写第一行代码。因为现在我越来越相信：

**好的 Web3 产品，不是因为用了区块链才成立。**

**而是因为区块链，让它变得更有意义。**

## 最后想说的

做这个项目的过程中，我经常在想一个问题：Web3 到底能给我们带来什么？

是更高的收益吗？是更炫的技术吗？是去中心化的意识形态吗？

可能都不是。对我来说，Web3 最迷人的地方在于——它让一些"没有实用价值"的东西有了存在的可能。一棵不能赚钱的树，在传统互联网里是找不到生存空间的，因为服务器要花钱，运营要成本，没有商业价值的东西活不下去。

但在区块链上，只要有一群人觉得它有意义，它就能一直存在下去。不需要公司来运营，不需要服务器来托管，它就静静躺在链上，和所有其他数据一起，成为这个时代的一部分。

这可能就是我为什么想做这棵树的原因。

它没有用，但它很美。

而美本身，就是意义。
